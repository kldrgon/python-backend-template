# 领域边界设计

## 目标

本文件用于说明初始化项目里如何划分领域边界，以及上下文之间如何协作。

## 基本原则

- 一个上下文只负责一组稳定的业务语义
- 聚合根是事务一致性边界
- 跨上下文只传 ID 或基础数据，不传领域对象
- 事件 payload 只放外部需要的最小信息

## 聚合规则

- 一次事务优先只修改一个聚合根
- 聚合内部实体由聚合根独占管理
- 跨聚合根引用优先用 ID
- 如果一个对象频繁跨边界共享，优先重新检查边界是否划错

## 上下文协作

常见方式：

- 同步调用：适合简单查询或轻量编排
- 领域事件：适合异步解耦
- 工作流编排：适合多步骤且需要补偿的流程

## 存储边界

- 业务域只保存业务需要的标识
- 文件访问由 Blob 上下文统一提供
- 业务表不要直接持久化 bucket、object_key、etag 等存储实现细节

## 代码落点建议

- `domain/`：领域概念、不变量、聚合行为
- `application/`：用例编排
- `adapter/input/`：输入适配
- `adapter/output/`：仓储、外部依赖实现
- `query/`：读侧查询

## 设计检查清单

- 这个能力是否属于独立业务边界
- 这个规则是否应落在聚合内
- 这个依赖是否属于外部能力
- 这个字段是否泄漏了存储或基础设施细节
