# Design Doc Reference

## Recommended Document Template

如果这份设计稿是本次过程中新建的临时设计文档，默认放在 `DEVELOP_TEMP/<timestamp>/`。

```text
# <需求名称> 设计稿

## 1. 目标
- 本次要解决什么问题
- 预期结果是什么

## 2. 范围
- 本次做什么
- 本次不做什么

## 3. 界限上下文划分
- 涉及哪些上下文
- 为什么这样划分
- 哪些职责不属于本上下文
- 每个上下文采用什么实现策略

## 4. 内部实现策略
- 哪些上下文采用领域模型
- 哪些上下文采用轻量实现
- 为什么这样选

## 5. Port 与外部依赖
- 本次涉及哪些跨上下文能力缺口
- 哪些依赖不能直接耦合

## 6. 数据流 / 事件流
- 命令链路
- 查询链路
- 是否存在事件、outbox、异步流程

## 7. 简化点与退化点
- 哪些上下文这次刻意不做重设计
- 为什么
- 以后何时需要升级

## 8. 待确认项
- 已确认
- 假设
- 待确认
```

## Bounded Context Heuristics

判断是否需要新界限上下文时，可先看：

- 是否有独立业务目标
- 是否有不同变化原因
- 是否有独立领域语言
- 是否会长期独立演进

如果只是已有上下文下的一组局部能力扩展，通常先不要拆新上下文。

## Modeling Depth Heuristics

适合采用领域模型的场景：

- 有明显不变量
- 有复杂状态转换
- 有跨步骤业务规则
- 需要通过行为约束保护一致性

适合采用轻量实现的场景：

- 主要是 CRUD
- 主要是查询拼装
- 规则薄
- 变化点主要在接口或展示，而不是领域行为

如果决定简化，应以界限上下文为单位整体简化，而不是在同一上下文内部零散混用重建模和轻量实现。

Port 命名与跨上下文依赖规则属于 `project-structure-and-rules`，设计稿里只需要说明本次有哪些跨上下文能力缺口。

## Simplification Examples

### 可以先轻量实现

```text
场景：
- 只需要一个 service 完成简单写入
- 没有明显聚合不变量
- 没有复杂对象转换

做法：
- 该上下文采用轻量实现
- application/service 直接编排
- 具体是否允许直接导入 model 写 SQL，回到 `project-structure-and-rules`
- 该上下文内部统一采用这套轻量策略，不再局部混入另一套主实现方式
```

### 应考虑补齐建模

```text
场景：
- 对象生命周期复杂
- 多种状态流转
- 有明确领域事件
- 后续会持续扩展业务规则

做法：
- 该上下文采用领域模型
- 明确核心领域对象、边界和不变量
- 再根据项目规则决定 repository、query side、异步链路等落点
```
