# Design Doc

## Use When

- 用户的需求还不完整，但已经需要形成可确认的设计稿
- 需要先落设计文档，再开始实现
- 需要决定界限上下文怎么划分
- 需要决定本次涉及到的每个界限上下文，是否采用领域模型
- 需要决定哪些上下文整体采用轻量实现

## Core Goal

本 skill 的目标不是把需求写成漂亮文档，而是把需求补全到“可确认、可落地、可约束后续实现”的状态。

输出结果应能回答三件事：

1. 这个需求到底要做什么，不做什么
2. 本次涉及到的每个界限上下文，分别采用什么实现策略，以及为什么
3. 后续实现时应受哪些边界与判断标准约束

## Design Principles For This Project

### 1. 先定界限上下文，再定实现细节

- 先判断是否需要新的界限上下文
- 先判断职责归属，再判断代码放置位置
- 如果边界没定，不直接展开实现细节

### 2. 设计决策的单位是界限上下文

- 不是整个需求统一选一种策略
- 应分别说明每个界限上下文是否采用领域模型
- 某个上下文采用领域模型，某个上下文采用轻量实现，允许并存
- 但同一个界限上下文内部不混用两套主实现策略

### 3. 先判断是否采用领域模型

- 业务规则强、状态转换复杂、存在明显不变量时，优先采用领域模型
- 如果主要是简单 CRUD、查询拼装、轻编排，也可以不采用领域模型，而使用轻量实现
- 一旦某个上下文选择采用领域模型，就要明确核心领域对象、边界和不变量

### 4. 轻量实现也要写清边界

- 不采用领域模型，不等于没有边界
- 即使采用轻量实现，也要说明职责归属、依赖方向、为何可以简化
- 轻量实现的具体规则，回到 `project-structure-and-rules`

### 5. Port 与依赖方式要写清楚

- 设计稿只需要说明本次涉及哪些跨上下文能力缺口
- 设计稿只需要说明哪些依赖不能直接耦合
- 具体 Port、Adaptor、领域模型、轻量实现规则，回到 `project-structure-and-rules`

### 6. 简化决策必须以界限上下文为单位

- 如果决定简化，应说明“哪个上下文整体采用轻量实现”
- 不写成零散功能点级别的随意简化

## Required Sections

设计稿至少写清下面这些：

1. 目标
   - 要解决什么问题
   - 预期结果是什么

2. 范围
   - 本次做什么
   - 本次不做什么

3. 业务边界
   - 涉及哪些界限上下文
   - 为什么这样划分
   - 哪些能力不归本上下文负责
   - 每个上下文采用什么实现策略

4. 内部实现策略
   - 哪些上下文采用领域模型
   - 哪些上下文采用轻量实现
   - 为什么这样选

5. 对外依赖与 Port
   - 本次涉及哪些跨上下文能力缺口
   - 哪些依赖不应直接耦合

6. 数据流或事件流
   - 命令链路
   - 查询链路
   - 是否存在事件、outbox、异步流程

7. 简化点与退化点
   - 哪些上下文刻意不做重设计
   - 为什么
   - 未来何时需要升级

8. 待确认项
   - 哪些结论已经确认
   - 哪些仍是假设
   - 哪些需要用户拍板

## Decision Checklist

写设计稿时，至少判断下面这些问题：

1. 这是新上下文，还是已有上下文扩展
2. 本次涉及哪些界限上下文
3. 哪些上下文需要采用领域模型
4. 哪些上下文可以整体采用轻量实现
5. 这里的 Port 是否真的是“能力抽象”，还是只是换了个名字的跨域耦合
6. 这是同上下文协作，还是跨上下文依赖
7. 是否存在“先轻量落地，后续再演进”的更优方案

## Required Behavior

- 如果边界不清，先补边界，不直接写实现细节。
- 如果用户需求还明显缺口，先把缺口显式写出来。
- 不把“是否采用领域模型”写成整个需求的一刀切结论，要按界限上下文分别说明。
- 不默认所有上下文都必须采用领域模型。
- 不允许同一界限上下文内部同时混用两套主实现策略。
- 如果是跨上下文依赖，就不要直接调用对方内部实现。
- 如果本次设计稿是临时工作文档，默认放在 `DEVELOP_TEMP/<timestamp>/`。
- 允许给出“当前先简化，未来再升级”的方案，但必须写清升级触发条件。

## Output Pattern

优先按这个格式产出设计稿：

```text
目标：
- <本次要解决什么问题>

范围：
- 做什么
- 不做什么

界限上下文：
- 涉及哪些上下文
- 为什么

各上下文实现策略：
- <上下文 A>：采用领域模型
- <上下文 B>：采用轻量实现
- 原因：<一句话>

Port 与依赖：
- 需要哪些跨上下文能力
- 哪些依赖不能直接耦合

简化点：
- 哪些上下文这次刻意不做重设计
- 为什么可以先不做

待确认：
- <需要用户确认的点>
```

## Handoff

设计稿产出后：

- 需要代码落点、命名、测试、提交规则时，转入 `project-structure-and-rules`
- 需要进入长流程实现时，转入 `branch-workflow`
